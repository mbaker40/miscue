`registry.ts` (as built):
```ts
export type Kind = 'player'|'bumper'|'skitter'|'charger'|'splitter'|'boss'; // FROZEN
export interface Entity {
  id: number; kind: Kind; elite: boolean;
  body: RAPIER.RigidBody; collider: RAPIER.Collider;
  alive: boolean; r: number;
  prevPos: { x; y; z }; prevRot: { x; y; z; w };
  spinSide: number; spinTop: number;
  ai: { timer; state; tx; ty; tz; lungesLeft; didSplit };
  armor: number; vulnerable: boolean;
}
export function spawnPlayer(sim: Sim, pos: Vec3): Entity;
export function spawnEnemy(sim: Sim, kind: Exclude<Kind,'player'>, pos: Vec3, elite?: boolean): Entity;
export function despawn(sim: Sim, e: Entity): void;
export function byCollider(handle: number): Entity | undefined;
export function all(): Entity[];
export function resetRegistry(): void;
```

`physics-events.ts` (as built — interface only; the concrete implementation is game-state wiring):
```ts
export interface PhysicsEvents {
  onSink(e: Entity, pocketId: number): void;
  onCollide(a: Entity, b: Entity, impact: number, tangent: number): void; // post-solve velocities (documented approximation)
  onSurface(e: Entity, impact: number, kind: 'wall' | 'floor'): void;     // |normal.y| > 0.6 → floor
  onPad(e: Entity, padId: number): void;
  onFall(e: Entity): void;   // fires once, on the downward crossing of killY (prevPos.y >= killY > y)
  onGraze(e: Entity, pocketId: number): void;
}
```

`sim.ts` (as built):
```ts
export async function initPhysics(): Promise<void>;  // RAPIER.init(), idempotent
export interface Vec3 { x: number; y: number; z: number }
export type Shape = { type:'cuboid'; hx; hy; hz } | { type:'cylinder'; halfHeight; radius };
export class Sim {
  world: RAPIER.World;  killY: number;               // defaults to CFG.killY
  addStatic(shape: Shape, pos: Vec3, rotDeg?: Vec3): number;
  addSensorZone(kind: 'pocket'|'pad', id: number, shape: Shape, pos: Vec3): number;
  snapshotPrev(entities: Entity[]): void;            // call BEFORE step — render lerps prev→current
  step(h: number, entities: Entity[], ev: PhysicsEvents): void;
  checkGraze(player: Entity, pockets: {x;y;z;id}[], ev: PhysicsEvents): void; // called by game loop, throttled 1/pocket/s
}
```
Notes: `onSurface` splits floor/wall by contact-normal `|n.y| > 0.6`; contact normals come from the
narrow-phase manifold with a center-to-center fallback; `sim.ts` holds no runtime dep on `registry.ts`
(entities are passed into `step`, type-only imports otherwise); ball↔sensor mapping lives in an internal
handle map. Graze is a proximity check (`checkGraze`), not a second sensor shell.

2. esbuild + node assertion (same recipe as v1's P1/P2 — bundle to the scratchpad, assert in node):

## M0-C — Control + input

**Files you own:** `v2/src/control.ts`, `v2/src/input.ts` (both new). Touch nothing else.
**Read first:** `docs/tasks-v2/README.md`, `docs/RACK_AND_RUIN_v2_dossier_v1.md` §05, `v2/src/config.ts` +
`v2/src/registry.ts` (from M0-B, already landed), and **v1's `src/input.ts`** — the spin pad and
double-tap-zero logic is ported near-verbatim from there, only the aim-vector math changes (rotated by camera
yaw instead of raw screen-to-table).

### Goal
Implement the physics-facing control functions (`fire`/`steer`/`grounded`) and the touch input zone model that
drives them.

### Current state
Neither file exists. `config.ts`/`registry.ts`/`sim.ts` exist from M0-B.

### Required behavior, exact numbers

`control.ts`:
```ts
export function fire(e: Entity, dirXZ: {x:number;z:number}, power: number, spin: {side:number; top:number}, stats: PlayerStatsLike): void;
export function steer(e: Entity, dirXZ: {x:number;z:number}, dt: number, stats: PlayerStatsLike): void;
export function grounded(e: Entity, sim: Sim): boolean;
```
- `fire`: normalize `dirXZ`, apply an impulse of magnitude `power * CFG.shot.maxImpulse * (stats.maxPower ??
  1)` along it to `e.body`. Set `spin.side`/`spin.top` as decaying scalars (stored on the entity's `ai` bag or
  a parallel map — your call, but keep it out of `registry.ts`'s frozen `Entity` shape). Under
  `CFG.spin.mode === 'forced'`: each physics step apply a lateral force perpendicular to velocity scaled by
  `spinSide * speed` (v1's `CURVE_K` constant, ported) and a forward/back force along velocity scaled by
  `spinTop` (v1's draw/follow constant), both decaying exponentially (v1's `SPIN_DECAY = 1.1/s`, ported).
  Under `'physical'`: stub is acceptable for M0 — apply the impulse as angular velocity + let friction do the
  curving; this mode is not the default and is not feel-tested until the M0/M1 spin decision lands.
- `steer`: no-op unless `CFG.steer.enabled` and `speedOf(e) >= CFG.steer.minSpeed`. Otherwise apply a force of
  magnitude up to `CFG.steer.accel * mass(e)` along `dirXZ`, and drain `stats.stroke -= CFG.steer.drainPerSec
  * dt` (clamped at 0) — this is the correction-only guarantee from the dossier, mechanically: because
  `steer.accel < g·sin(9°)`, this force can nudge a line but never climb a ramp or start the ball from rest.
- `grounded`: raycast straight down from `e`'s position, length `e.r + 0.05`, against static colliders only;
  `true` if it hits within that distance.
- Powerup hooks (`hop`, `slowFall`, `timeBrush`) — stub function signatures only for M0 (they're implemented
  at M4); presence matters more than behavior right now, so later packets don't need to touch `control.ts`'s
  public shape to add them.

`input.ts` — touch zone model, ported from v1's pointer-event structure:
- Lower ~55% of screen height, drag = slingshot aim. The raw screen drag vector must be **rotated by the
  current camera yaw** before becoming `dirXZ` — but `input.ts` must **not** import `camera.ts` directly
  (that file is a parallel M0-D packet in the same dispatch batch). Instead, `input.ts`'s constructor takes an
  injected `getCameraYaw: () => number` callback; `main.ts` (M0-D) wires the real camera in. This is the one
  cross-packet seam in this batch — if it looks wrong, it's still correct: it's how M0-C and M0-D stay
  disjoint while dispatched in parallel.
- Upper zone, or any two-finger drag = manual orbit signal (expose it as a callback/event, `main.ts` forwards
  it to `camera.ts`).
- The same lower-zone drag, while the ball is coasting (aim ineligible), reads as steering instead of aim.
- Aim eligibility gate: `canAim: (() => boolean) | null`, same shape as v1 — wired to `grounded(e, sim) &&
  speedOf(e) < someSlowThreshold` by `main.ts`.
- Spin pad + double-tap-zero: port v1's `src/input.ts` pad logic verbatim (draggable knob, `dx → spinSide`,
  `-dy → spinTop`, clamp to unit circle, 300 ms double-tap zeroes both and snaps the knob to center).
- Peek button (`#peek`) and powerup slot tap (`#slot`) each expose a simple callback (`onPeekToggle`,
  `onSlotTap`) for `main.ts` to wire.

### Implementation steps, file by file
1. `control.ts` — `fire`/`steer`/`grounded` plus powerup-hook stubs, per the contract above.
2. `input.ts` — pointer-event wiring for the aim/steer/orbit/spin-pad/peek/slot zones, with the
   `getCameraYaw` injection point and the callback surface `main.ts` needs.

### Interfaces that must not break
`fire`/`steer`/`grounded` signatures and argument order; `input.ts`'s constructor accepting
`getCameraYaw: () => number` (not importing `camera.ts`); the fired-shot payload shape mirrors v1's
`ShotPayload`: `{ dirX, dirZ, power, spin?: { side, top } }`.

### Out of scope
No camera implementation (that's M0-D), no rendering, no HUD wiring beyond the callback surface above, no
powerup effects beyond the stub signatures.

### Acceptance checks
1. `npx tsc --noEmit` and `npm run build`, both inside `v2/`, pass.
2. Headless Playwright (this batch's runtime assertion is joint with M0-D, since `__game.debug` is wired
   there — same pattern as v1's P3/P4 typechecked-together batch): once M0-D lands, `debug.fire(...)` moves
   the ball, and calling `steer(...)` (directly, or via a debug hook) while the ball is at rest produces no

`v2/src/config.ts`/`registry.ts`/`sim.ts` (from M0-B, already landed).

### Goal
Stand up a minimal three.js scene, the chase-camera rig, one hand-built level, and the boot sequence —
including the frozen `window.__game.debug` hook every later milestone's headless verification depends on.

### Current state
`v2/src/main.ts` is a placeholder that renders static text into `#app` and does nothing else. No camera,
scene, or level files exist.

### Required behavior, exact numbers

`camera.ts` — a chase rig:
- Critically-damped spring toward a desired offset (`CFG.cam.dist` behind, `CFG.cam.height` above the ball),
  using `CFG.cam.spring` as the spring constant.
- Yaw auto-tracks the ball's velocity heading while moving; when the ball is parked (not moving / aim-ineligible), yaw is driven by manual orbit input instead (wired from `input.ts`'s orbit callback via `main.ts`).
- Look-ahead: bias the look target `CFG.cam.lookAhead` units along the ball's current velocity direction.
- Occlusion pull-in: raycast from the look target back toward the desired camera position; if a static
  collider blocks it, pull the camera in along that ray instead of clipping through geometry.
- `peek(on: boolean)`: lerp to an overhead framing (looking straight down from roughly `CFG.cam.peekHeight`)
  over **0.25 s**, preserving the current yaw so the player's sense of facing carries through; toggling off
  lerps back the same way.
- Exposes a `yaw` getter — this is what `input.ts`'s injected `getCameraYaw` callback reads.

`render/scene.ts` — `THREE.Scene`, a `WebGLRenderer` with device pixel ratio capped at `CFG.fx.dprCap`, basic
fog for the void, minimal lighting (directional + ambient is enough for M0 — full bloom/composer work is
later milestones' `render/fx.ts`), and a resize handler.

`render/balls.ts` — one mesh per `Entity` of kind `player`/enemy kinds present: sphere geometry at radius
`CFG.ballR`, plain (non-textured) materials are acceptable for M0 — v1's texture/bloom treatment is ported in
a later milestone's full render split, not this one. Position/quaternion each frame come from lerping
`prevPos → pos` and slerping `prevRot → rot` by the render-alpha (see the accumulator loop below) — this
replaces v1's manual roll math entirely; Rapier's real rotation *is* the roll now.

`level0.ts` — `buildLevel0(sim: Sim): { spawn: Vec3; checkpoint: Vec3 }`: one hand-built static platform
within the platform-size convention (6–10 logical units), consisting of:
- a flat slab to spawn on and return to,
- a ramp segment (any reasonable authored angle, e.g. 15–20° — well above the ~9° steering-cannot-climb
  ceiling, so *rolling* up it under momentum works fine, only *steering alone* cannot start a climb),
- a bank wall (a vertical collider you can carom a shot off),
- a gap with nothing beneath it down to `CFG.killY`, so falling in actually triggers `onFall`.
Register a spawn point on the slab and a checkpoint (can be the same point for M0 — checkpoints proper are an
M3 concept).

`main.ts` (full replacement of the stub):
1. Dynamic-import Rapier, `await initPhysics()`.
2. Construct `Sim`, call `level0.buildLevel0(sim)`, `sim.addBall(spawn, 'player')`.
3. Construct `camera.ts`'s rig, `render/scene.ts`'s scene/renderer, `render/balls.ts`'s ball meshes.
4. Construct `control.ts`'s functions bound to the player entity, and `input.ts`, injecting
   `getCameraYaw: () => camera.yaw` and wiring orbit/peek/slot callbacks to the camera and (stub) slot state.
5. Wire a minimal concrete `PhysicsEvents` implementation: at minimum, `onFall(e)` should respawn that entity
   at the checkpoint (`body.setTranslation`/`setLinvel(0)` — a "warp + settle", not yet the full Stroke/crack
   economy, which is M1+ wiring). Other seam methods can no-op for M0.
6. Fixed-step accumulator loop: accumulate real `dt`, call `sim.snapshotPrev()` then `sim.step(CFG.h, events)`
   in a `while (accumulator >= CFG.h)` loop, then render with `renderAlpha = accumulator / CFG.h`.
7. Toggle `#hud`'s `.on` class so the HUD container is visible (full widget wiring is later milestones' job;
   the container should not stay hidden after M0).
8. Expose `window.__game` and the frozen debug hook:
   ```ts
   window.__game = {
     debug: {
       state() { /* return a minimal snapshot: ball position, grounded, etc */ },
       fire(dirXZ, power, spin?) { /* calls control.fire on the player entity */ },
       warp(x, y, z) { /* teleports the player entity, zeroes velocity */ },
       settle() { /* zeroes velocity/angvel without moving position */ },
       enterNode(kind) { /* stub for M0 — route/nodes don't exist yet; no-op or throw a clear "not until M2" */ },
       grantPowerup(p) { /* stub for M0 — no-op; powerups land at M4 */ },
       tick(ms) { /* advance the fixed-step loop by ms of simulated time, for deterministic test drives */ },
     },
   };
   ```
   `enterNode`/`grantPowerup` must exist with these exact names/arities even as stubs — later milestones fill
   them in, they never get renamed.
